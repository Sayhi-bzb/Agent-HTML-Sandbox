# Best Practice

Write artifacts for people who need to read, compare, decide, and respond. agent-html is not a prettier Markdown wrapper; it is a stable HTML surface for agent work that would otherwise become a long, fragile Markdown file.

## Check the managed runtime [#check-the-managed-runtime]

Normal commands bootstrap the user-level managed runtime automatically. Use checks before debugging:

```bash
ahtml doctor
```

Runtime files live under `~/.ahtml` or `%USERPROFILE%\.ahtml` by default. Set `AHTML_HOME` only when you need an isolated runtime for tests or debugging.

## Read schema first [#read-schema-first]

Start every generation task with `ahtml prompt`. The schema prompt is the only public contract for components and config.
Use the approved document style config choices from the schema as the only public visual choice. Author them through `style-ref` values, not free-form `theme`, `density`, `tone`, or `width` values in agent-facing input.

## Use semantic components [#use-semantic-components]

The agent should describe information structure, not page implementation. Pick components by the role they play in the artifact:

- `card` for grouped content.
- `alert` for important notes.
- `table` for structured evidence.
- `list` for ordered or unordered points.
- `tabs` and `accordion` when the artifact needs multiple readable views or expandable sections.

## Keep structure simple [#keep-structure-simple]

Prefer a clear hierarchy:

```txt
page
  card
    table / list / alert / text
```

Use deeper structures only when the content genuinely needs sections, tabs, or expandable groups.

## Do not write implementation details [#do-not-write-implementation-details]

The HTML artifact should feel stable across runs. Keep low-level implementation details out of the agent-facing document so the renderer can preserve visual consistency, safety, and portability.

Never use these in agent-facing documents:

```txt
class
className
style
css
Tailwind classes
script
onclick
event handlers
Radix props
shadcn props
external URLs
raw HTML passthrough
unknown HTML attributes
```

agent-html is not a raw HTML escape hatch.

## Prefer HTML-worthy outputs [#prefer-html-worthy-outputs]

Use agent-html when the result benefits from more than a wall of Markdown:

- Planning and specification artifacts.
- PR and code-review explainers.
- Research reports and incident reports.
- Decision records with evidence.
- Feedback artifacts that return choices or notes to the agent.

## Repair validation errors [#repair-validation-errors]

If validation fails, fix the document instead of bypassing the error.

- Unknown component: use a registered standard component.
- Unknown attr: remove the prop or use the schema name.
- Invalid child: move the component into an allowed parent.
- Invalid render config: use only listed document style config choices and allowed semantic props.
- Missing root: wrap content in `<page title="...">`.

## Debug setup before editing artifacts [#debug-setup-before-editing-artifacts]

Use setup checks before changing documents:

```bash
ahtml doctor
```

- If runtime checks fail, rerun `ahtml doctor`; it repairs the managed runtime automatically.
- If the package is outdated, follow the update command printed by `doctor`.
- If validation fails, repair the agent-html document instead of bypassing validation or changing runtime behavior.
