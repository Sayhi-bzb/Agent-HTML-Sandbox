# Getting Started with agent-html

agent-html helps agents replace long Markdown outputs with stable, shareable HTML artifacts.

Markdown is a good scratchpad, but it is a weak final surface for dense agent work. Plans, PR explainers, research reports, design comparisons, decisions, and feedback loops need visual hierarchy, tables, controlled interaction, and a result people can open and share.

agent-html gives agents an AI-native authoring format for that job. The agent writes semantic `.agent.html`; `ahtml` validates it against a stable schema and renders a portable HTML artifact. The final artifact is for humans to read, compare, share, archive, and hand back into the agent loop.

<Callout title="Primary rule">
  Read the schema first so the agent writes semantic artifact structure instead
  of raw HTML, CSS, JavaScript, Tailwind classes, event handlers, React props,
  Radix props, or shadcn props.
</Callout>

## 1. Install the CLI [#install-the-cli]

`@agent-html/ahtml` is published once to npmjs.com. Pick your package manager:

<CodeBlockTabs defaultValue="npm">
  <CodeBlockTabsList>
    <CodeBlockTabsTrigger value="npm">npm</CodeBlockTabsTrigger>
    <CodeBlockTabsTrigger value="pnpm">pnpm</CodeBlockTabsTrigger>
    <CodeBlockTabsTrigger value="yarn">yarn</CodeBlockTabsTrigger>
    <CodeBlockTabsTrigger value="bun">bun</CodeBlockTabsTrigger>
  </CodeBlockTabsList>

  <CodeBlockTab value="npm">

```bash
npm install -g @agent-html/ahtml
ahtml setup
ahtml status
ahtml doctor
```

  </CodeBlockTab>

  <CodeBlockTab value="pnpm">

```bash
pnpm add -g @agent-html/ahtml

ahtml status
ahtml doctor
```

  </CodeBlockTab>

  <CodeBlockTab value="yarn">

```bash
yarn global add @agent-html/ahtml
ahtml setup
ahtml status
ahtml doctor
```

  </CodeBlockTab>

  <CodeBlockTab value="bun">

```bash
bun add -g @agent-html/ahtml
ahtml setup
ahtml status
ahtml doctor
```

  </CodeBlockTab>
</CodeBlockTabs>

`ahtml setup` configures the managed runtime under `~/.ahtml`, currently with a shadcn-backed renderer. `ahtml status`, `ahtml doctor`, `ahtml build`, and `ahtml preview` initialize or repair the managed shadcn runtime when setup has not run. They do not write `src/`, `components.json`, `vite.config.ts`, or `agent-html.project.json` into the current repository.

`ahtml status` and `ahtml doctor` can show a non-blocking package update hint. Set `AHTML_NO_UPDATE_CHECK=1` to disable that check.

## 2. Read the schema [#read-the-schema]

```bash
ahtml schema --format prompt
```

Use the schema output to learn the registered components, allowed props, allowed children, and approved presentation profile values.

## 3. Write a document [#write-a-document]

A valid document needs a `<page>` root.

```html title="artifact.agent.html"
<meta-agent profile="report-default" />

<page title="Review">
  <card title="Summary">
    This review is a stable HTML artifact instead of a long Markdown note.
  </card>
</page>
```

## 4. Validate before building [#validate-before-building]

```bash
ahtml validate --input artifact.agent.html
```

Validation catches unknown components, unknown props, blocked implementation details, invalid children, and invalid presentation profile values before artifact generation.

## 5. Build the artifact [#build-the-artifact]

```bash
ahtml build --input artifact.agent.html --out dist/html
```

The output is a static artifact directory that can be opened, shared, deployed, or archived.

## 6. Preview locally [#preview-locally]

```bash
ahtml preview --input artifact.agent.html --out dist/html --port 4173
```

Preview serves the same static output produced by `build`. It does not use a separate renderer.

## Inspect the result [#inspect-the-result]

```bash
ahtml inspect --input artifact.agent.html
```

Use `inspect` to review effective config and component usage without inspecting runtime files.
