## Product north star [#product-north-star]

agent-html exists to replace long Markdown agent outputs with AI-native HTML artifacts. Development decisions should protect that product shape: agents write semantic `.agent.html`, users receive stable shareable HTML, and implementation details stay behind the schema and managed runtime.

## Repository development [#repository-development]

```bash
npm install
npm link --workspace @agent-html/ahtml
ahtml doctor
ahtml prompt
```

Repository fallback:

```bash
npm run agent -- prompt
```

Local check levels:

```bash
npm run build
npm run test:run -- <file-or-pattern>
npm run test:run:cli-heavy
npm run check:package
npm run check:package:release
npm run check:docs:ci
npm run check:docs
npm run check:all
```

Follow `AGENTS.md` for verification cadence. This page lists available gates; it does not redefine when to run them.

While editing, prefer the narrowest useful check:

- For pure governance docs or ordinary copy edits, inspect the final diff only.
- For docs changes that alter MDX structure, examples, routes, generated output, or docs build behavior, run the smallest relevant docs gate.
- For ordinary CLI/runtime code changes, run `npm run build` once, then `npm run test:run -- <affected test file or -t pattern>`.
- Run `npm run test:run:cli-heavy` only when changes touch setup, doctor, prompt output, build, preview, or runtime rendering.
- Use `npm run check:package` as the package CI gate, not as the default edit loop.
- Use `npm run check:package:release` only for release-boundary validation. It adds `verify:pack` on top of the package CI checks.

`dev:check`, `check:quick`, and `check:ready` remain broad compatibility aliases in `package.json`, but they are not the preferred day-to-day edit loop.

## Integrate with an app [#integrate-with-an-app]

If you build a product app around `ahtml`, keep the app in a separate repository unless a future spec explicitly changes the package boundary.

Recommended integration order:

1. Consume the published npm package `@agent-html/ahtml`.
2. Import `@agent-html/core` only when you explicitly need the shared schema or sanitization library surface.
3. For local development, use `npm pack` or `npm link --workspace @agent-html/ahtml`.
4. Do not import `packages/ahtml/src/*` or `packages/core/src/*` across repositories.

Prefer stable CLI boundaries over cross-repo source imports:

```bash
ahtml prompt --format json
ahtml build artifact.agent.html --format json
ahtml inspect --input artifact.agent.html --format json
ahtml doctor --format json
```

Use the CLI exit code as the success signal. In JSON mode, `build`, `inspect`, `prompt`, and `doctor` print machine-readable output to stdout for app-side orchestration.

## Verify package boundary [#verify-package-boundary]

```bash
npm pack --dry-run --workspace @agent-html/core
npm pack --dry-run --workspace @agent-html/ahtml
```

The workspace publishes two packages: `@agent-html/core` and `@agent-html/ahtml`. `@agent-html/core` should include only shared contract and sanitization code. `@agent-html/ahtml` should include only the CLI, runtime config, bin entry, managed runtime template logic, and README. Neither package should include docs web files, examples, specs, blueprints, or tests.

## Verify installed CLI behavior [#verify-installed-cli-behavior]

```bash
npm run verify:pack
```

This packs the project, installs it into a temporary consumer directory, sets `AHTML_HOME` to an isolated runtime directory, and verifies installed `ahtml` commands without creating project scaffold files.

## Verify published package [#verify-published-package]

```bash
npm view @agent-html/core dist-tags versions --json
npm view @agent-html/ahtml dist-tags versions --json
npm install -g @agent-html/ahtml
```

The release workflow targets two npm packages: `@agent-html/core` and `@agent-html/ahtml`; the CLI bin remains `ahtml` through `@agent-html/ahtml`. Verify actual registry state with `npm view` before assuming both packages are already available on npmjs.com. pnpm, yarn, and bun consume the same npm registry packages and do not need separate package releases.

## Publish npm package [#publish-npm-package]

Publishing is tag-driven through GitHub Actions and npm Trusted Publishing. Do not use a long-lived `NPM_TOKEN` for releases.

Configure the npm trusted publisher for both `@agent-html/core` and `@agent-html/ahtml`:

```txt
Provider: GitHub Actions
Owner: Sayhi-bzb
Repository: Agent-HTML
Workflow filename: publish-npm.yml
```

Release a package version:

```bash
npm version patch --workspace @agent-html/core --workspace @agent-html/ahtml --no-git-tag-version
# or: npm version minor --workspace @agent-html/core --workspace @agent-html/ahtml --no-git-tag-version
# or: npm version major --workspace @agent-html/core --workspace @agent-html/ahtml --no-git-tag-version
```

The release version source is the pair of workspace package manifests:

- `packages/core/package.json`
- `packages/ahtml/package.json`

These two versions must stay identical. The root workspace version is not the release source of truth.

Validate and publish the selected version:

```bash
npm run release:check
git add -A
git commit -m "chore: release <version>"
npm run release:tag
git push origin main --tags
```

The publish workflow fails if the Git tag does not match both workspace package versions.

Supported release channels:

- Stable: `0.2.0` -> Git tag `v0.2.0` -> npm dist-tag `latest`
- Alpha: `0.2.0-alpha.1` -> Git tag `v0.2.0-alpha.1` -> npm dist-tag `alpha`

Release a prerelease version:

```bash
npm version prerelease --preid alpha --workspace @agent-html/core --workspace @agent-html/ahtml --no-git-tag-version
```

Validate by channel:

```bash
npm run release:check:stable
npm run release:check:alpha
```

Install the latest stable release:

```bash
npm install -g @agent-html/ahtml
```

Install the current alpha release:

```bash
npm install -g @agent-html/ahtml@alpha
```

`release:check` is the generic local package release gate. It runs the package CI checks, `verify:pack`, and the npm dry-run publish for both workspace packages using the dist-tag derived from the current workspace version. The GitHub Actions publish workflow uses the same version-derived dist-tag before tag-driven publish.

## Distribute the ahtml skill [#distribute-the-ahtml-skill]

The `ahtml` agent skill lives at `.agents/skills/ahtml`. The open `skills` CLI discovers `.agents/skills/` automatically.

List skills from this repository:

```bash
npx skills add Sayhi-bzb/Agent-HTML --list
```

Install the `ahtml` skill:

```bash
npx skills add Sayhi-bzb/Agent-HTML --skill ahtml
```

Test installation from a local checkout:

```bash
npx skills add . --skill ahtml -a codex --copy -y
```

## Runtime architecture [#runtime-architecture]

```txt
installed ahtml package
  -> depends on @agent-html/core
  -> CLI + runtime config
  -> shadcn-backed managed runtime under ~/.ahtml or AHTML_HOME
  -> static artifact directory
```

`ahtml build` must use the managed runtime. The runtime may contain Vite, React, and shadcn-backed renderer files under `~/.ahtml/runtime`, but the current project must not receive that scaffold. Do not restore current-directory project integration, `agent-html.project.json`, `init --scaffold`, package-local Vite apps, package-local renderer apps, or root shadcn UI files.

Guided setup reads shadcn component catalogs, preset names, and preset code validation from shadcn package APIs, then uses shadcn CLI inside the managed runtime when components need to be installed. Do not maintain parallel shadcn component lists, preset lists, or registry schemas in ahtml docs, schema, or package source.

## Maintenance checklist [#maintenance-checklist]

Recurring verification commands live in `package.json`, and the repository gates are enforced by `.github/workflows/ci.yml`, `deploy-docs.yml`, and `publish-npm.yml`.

Use direct references instead of maintaining parallel facts:

- Verification gate names live in `package.json`; CI/release usage lives in `.github/workflows/`.
- Verification timing and scope live in `AGENTS.md`.
- Agent-facing component and prop choices live in `ahtml prompt`, derived from the shared public agent contract and runtime contract builders.
- Renderable component capability and runtime export requirements live in `packages/ahtml/src/config/component-capabilities.mjs` and `packages/ahtml/src/config/render-capabilities.mjs`.
- Managed runtime completeness checks live in `packages/ahtml/src/cli/runtime-surface.mjs` and `ahtml doctor`.
